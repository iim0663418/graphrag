# BDD 規格：修復視覺網絡細節問題

## Scenario 1: 計算並返回最大連接度 (max_degree)
**Given**: GraphRAG 關係數據包含實體的連接信息
**When**: 前端請求 GET /api/statistics
**Then**: 
- 後端計算每個實體的連接度 (degree)
- 返回最大連接度值 `max_degree`
- 前端統計面板顯示真實的最大連接度而非 0

## Scenario 2: 計算並返回社群活躍度
**Given**: GraphRAG 社群數據包含實體數量和關係信息
**When**: 前端請求 GET /api/communities  
**Then**:
- 後端計算每個社群的活躍度指標 (基於實體數量和內部關係密度)
- 返回 `activity` 欄位而非 N/A
- 前端社群面板顯示數值化的活躍度

## Scenario 3: 實現探索社群功能
**Given**: 用戶點擊社群分析面板的"探索社群"按鈕
**When**: 觸發探索社群事件
**Then**:
- 顯示該社群的詳細信息模態框或跳轉
- 包含社群內實體列表、關係圖譜、詳細分析
- 提供有意義的互動功能

## Scenario 4: 動態節點語義描述
**Given**: 用戶在知識圖譜中選擇了特定節點
**When**: 顯示節點關聯脈絡信息
**Then**:
- 基於實際節點數據生成語義描述
- 包含節點類型、連接數、相關實體等真實信息
- 移除固定的"語義嵌入空間"等通用文字

## 技術要求
- 最小化實作，只添加必要的計算邏輯
- 使用現有 parquet 數據進行計算
- 保持 API 響應性能
- 確保前端正確顯示動態數據
